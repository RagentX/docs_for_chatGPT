Формирование сообщения электронной почты

Show/Hide Hidden Text

Сформировать сообщение для отправки на электронную почту должника можно для всех долгов, кроме тех, которые находятся в статусах 6 "Завершено", 7 "Аннулировано", 8 "Отозван клиентом".

       Учет ограничений коммуникации

Проверка наличия схемы коммуникации

1.Проверяется значение константы CommunicationLimit.ContactTypeList:

·Если константа содержит код 8 (см. словарь № 3), то для коммуникации типа "Email" проверка наличия схемы коммуникации продолжается;

·Если данный код в константе отсутствует, то проверка не осуществляется, и коммуникация разрешается.

2.Проверяется в шаблоне сообщения флаг Учитывать в ограничениях (для звонков и выездов — см. Форма настройки исхода контакта в модуле Настройка дерева контактов; для SMS — см. SMS-сообщение; для email — см. Email-сообщение в модуле Сценарии сообщений):

·Если флаг включен, то проверка продолжается;

·Если флаг выключен, то проверка завершается, и коммуникация разрешается.

3.Проверяются Персональные ограничения коммуникации, настроенные для данного долга в Карточка должника.

·Если персональные ограничения настроены, то для данного типа контакта проверяется наличие Схемы коммуникации и соответствие даты совершения коммуникации ее периоду. Если ограничения настроены корректно, то осуществляется проверка лимита коммуникации по данной схеме (см. далее).

·Если персональные ограничения не настроены, или настроены некорректно или период не включает текущую дату, то проверка движется дальше.

4.Проверяются общие ограничения, налагаемые по формуле, указанной в значении константы, соответствующей данному типу коммуникации.

·Если в формуле указан существующий идентификатор Схемы коммуникации и дата совершения коммуникации входит в ее период, то осуществляется проверка лимита коммуникации по данной схеме (см. далее).

·Если общие ограничения не настроены, настроены некорректно или не включают текущую дату, коммуникация разрешается.

       Список констант для общих ограничений коммуникации

Для каждого типа коммуникации соответствующая константа содержит идентификатор формулы. по которой рассчитывается ограничение коммуникации в зависимости от параметров долга, участника договора, его роли и др. Результатом работы формулы является число — идентификатор той схемы коммуникации, которая должна быть использована в данном случае.

·Для звонка (набор номера) — CommunicationLimit.Call.Formula;

·Для SMS — CommunicationLimit.SMS.Formula;

·Для Email — CommunicationLimit.Email.Formula;

·Для выезда — CommunicationLimit.Visit.Formula;

·Для автоинформатора — CommunicationLimit.AutoInfo.Formula.

Проверка лимита коммуникации по схеме коммуникации

Процедура проверки лимита коммуникации осуществляет проверку возможности совершения коммуникации согласно текущей статистике для конкретного участника договора.

Если коммуникация разрешена, то она осуществляется. Если коммуникация превышает разрешенный лимит, то пользователю показывается соответствующее сообщение, и коммуникация не осуществляется.

Примечание. Для контактов типа SMS и Email отправка сообщения со свободным текстом не подвергается ограничениям.

       Возможность осуществления коммуникации при превышении лимита

Существует ряд доступов, дающих пользователю возможность осуществить коммуникацию сверх лимита.

Если в роли пользователя для данного типа коммуникации такой доступ включен, то при превышении лимита ему выдается соответствующее сообщение:

Формирование сообщения превысит допустимый лимит коммуникации. Вы действительно хотите <название действия для данного типа коммуникации>?

Выберите:

·Yes — если вы хотите совершить коммуникацию сверх лимита;

·No — если вы не хотите совершить коммуникацию сверх лимита.

Доступы на превышение лимитов коммуникации разных типов

·Доступ № 633 COMMUNICATION_SMS_EXCEED_LIMIT — дает право на формирование SMS – сообщений при превышении лимита согласно схеме коммуникации;

·Доступ № 634 COMMUNICATION_EMAIL_EXCEED_LIMIT — дает право на формирование EMAIL – сообщений при превышении лимита согласно схеме коммуникации;

·Доступ № 635 COMMUNICATION_CALL_EXCEED_LIMIT — дает право на совершение исходящего вызова при превышении лимита коммуникации;

·Доступ № 636 COMMUNICATION_VISIT_EXCEED_LIMIT — дает право на подготовку выезда при превышении лимита коммуникации.

 

Возможность формировать сообщения электронной почты из модуля "Карточка должника" определяется доступом DEBT_CARD_EMAIL. Возможность формировать сообщения электронной почты со свободным текстом определяется доступом DEBT_CARD_EMAIL_TEXT.

Примечание. Доступен функционал прикрепления вложений к сообщениям электронной почты. Данный функционал включается по согласованию с вендором. Для включения функционала прикрепления вложений к сообщениям электронной почты обратитесь в поддержку. Ф-л включается технической константой DebtEmail.Attachments.Use. Если в вашей версии Системы функционал включен — см. описание Прикрепление и просмотр вложений к сообщениям электронной почты.

Примечание. Если у текущего пользователя в роли значение доступа 511 DEBT_CARD_EMAIL_NOTREGISTERED имеет значение TRUE, то ему разрешено вручную вводить адрес в поле Адрес электронной почты. Кроме того, он может открывать форму редактирования нового сообщения и в тех случаях, когда у должника отсутствует зарегистрированный адрес электронной почты.

Для того чтобы сформировать сообщение, выполните следующие действия:

1.На вкладке Общая перейдите на вложенную вкладку Email, выберите адрес электронной почты, нажмите кнопку и в раскрывшемся списке выберите пункт Сформировать Email. Откроется форма редактирования нового сообщения.

2.В открывшейся форме сформируйте текст сообщения одним из следующих способов:

·на вкладке Текст введите текст (до 4096 знаков) и тему сообщения в соответствующие поля;

·на вкладке Сценарий в поле EMAIL-сообщение выберите шаблон сообщения. Текст сообщения, сформированный по выбранному шаблону, будет отображаться в поле Текст сообщения. Тему сообщения укажите вручную в поле Тема Email-сообщения. Если выбран шаблон, который содержит переменные, относящиеся к акции, то в форме будет активно поле Акция. В нем можно выбрать одну из активных акций, чтобы предложить ее клиенту. При отправке сообщения для долга (если он участвует в акции) будет установлен флаг Акция предложена и автоматически заполнено поле Источник информации об акции.

Примечание. Доступ для пользователя к созданным шаблонам настраивается в модуле Роли и доступы на вкладке Управление доступом к шаблонам Email-сообщений. В списке отображаются сообщения, доступ к которым был настроен.

Примечание. Если константа Email.OriginatorSelect имеет значение "Истина", то доступна возможность выбора отправителя при формировании EMAIL-сообщения. Список возможных значений поля задается в словаре № 300 "Отправитель EMAIL". При этом по умолчанию в поле "Отправитель" вводится термин словаря, указанный в значении константы Email.DefaultOriginator.

Примечание: шаблоны сообщений электронной почты формируются в модуле "Сценарии сообщений" (см. Создание сценария сообщения электронной почты). В списке вариантов для выбора будут доступны только те сценарии, для которых установлен флаг Единичная рассылка.

 Примечание для Поддержки. 

Лето банк, интеграция с WINGS (смс и email) — см. Интеграция с WINGS.

3.Нажмите кнопку . Заявка на отправку сообщения электронной почты будет сформирована и доступна для последующей обработки в модуле "Исходящее информирование" (см. Обработка заявок на отправку сообщений электронной почты).
Можно также отправить сообщение электронной почты с помощью приложения Microsoft Outlook. Для этого нажмите кнопку . Созданное сообщение откроется для редактирования и отправки в приложении Microsoft Outlook. Если на компьютере пользователя не установлено указанное приложение, будет выдано соответствующее сообщение.

       Примечание для поддержки. Ф-л ограничения коммуникации. Учет статистики

Учет статистики по Email

Статистика по email должна обновляться сразу после вставки сообщения в таблицу debt_email. В статистике будут учитываться сообщения в следующих статусах: 

·Заявка 

·Доставлено

Если сообщение будет переведено в статус, отличный от описанных выше, то оно будет исключено из статистики, и наоборот, если сообщение перейдет в указанные статусы, то оно будет учитываться в статистике. 

В качестве даты сообщении для учета в статистике берется фактическая дата отправки (debt_email.send_date), если эта дата не заполняется, то учитывается дата начала отправки (debt_email.start_date). Обновление статистки по email, которые изменили статус будет выполняться фоновым заданием.
