Смена статуса задачи (код 32)

См. далее — Логика работы мероприятия.

Порядок действий

1.Выберите тип мероприятия — Смена статуса задачи (код 32). Откроется форма Мероприятие для данного типа мероприятий.

2.Выберите группу объектов (объект "Задачи"), для обработки которых предназначено мероприятие.

3.Настройте параметры мероприятия:

·Статус — статус, который будет присвоен задачам. Значения выбираются из словаря № 275. Поле обязательно для заполнения;

·Причина смены статуса — Поле доступно только при выборе статусов, подразумевающих причину смены. Для каждого статуса причина смены выбирается из соответствующего словаря. Поле не обязательно для заполнения.

·Получатели уведомления — список получателей уведомления по словарю № 322 Получатели уведомлений по задачам. Для каждого получателя возможно указать необходимость отправки уведомления как самому пользователю, так и его Руководителю / Зам. Руководителя / Куратору. Список оформлен как таблица с возможностью множественного выбора. Поле не обязательно для заполнения.

4.Настройте График запуска мероприятия.

5.Заполните поля:

·Последовательность — при необходимости введите числовое значение последовательности. В случае одновременного запуска двух и более мероприятий первым будет выполняться задание с наименьшим значением в поле.

·Флаг "Вкл." — если флаг находится в положении Включено, мероприятие запускается, если флаг находится в положении Выключено, мероприятие не запускается. Использование флага дает возможность отменить запуск мероприятия, не удаляя его.

·Флаг "Проверять формирование групп". При включенном флаге мероприятие выполняется только для тех групп, которые были сформированы или переформированы в соответствии со следующими условиями:

·Если значение константы Scheduler.ReformGroupCheck.Mode равно 2, то группа считается переформированной, если дата формирования больше или равна текущей дате.

·Если значение константы Scheduler.ReformGroupCheck.Mode равно 1, то группа считается переформированной, если дата формирования больше текущей даты и времени минус 1 сутки (текущая логика).

·Если значение константы Scheduler.ReformGroupCheck.Mode не равно 1 или 2, то используется логика режима 1.

Внимание! Если мероприятие в модуле Расписание не предполагает использование автогруппы, флаг Проверять формирование группы должен находиться в положении Выключено.

·Комментарий — дополнительная информация.

6.Нажмите OK.

 

Логика работы мероприятия

В результате выполнения мероприятия всем задачам из выбранной группы устанавливается выбранный статус. Если указана причина смены статуса, она также сохраняется в задаче.

Выбор получателей

Если список получателей уведомлений не пуст, то формируется сообщение на внутреннюю почту об изменении статуса задачи всем отмеченным получателям:

·Пользователь:

·Постановщик — оператор, поставивший задачу;

·Плановый исполнитель — оператор, назначенный исполнителем задачи;

·Выполнивший задачу — оператор, выполнивший задачу;

·Ответственный — оператор, ответственный за долг (если задача поставлена по долгу) или за дело судебной работы или ИП (если задача поставлена соответственно по делу судебной работы или ИП);

·Руководитель — руководитель в подразделении пользователя (согласно структуре предприятия);

·Зам. Руководителя — заместитель руководителя в подразделении пользователя (согласно структуре предприятия);

·Куратор — куратор в подразделении пользователя (согласно структуре предприятия).

Создание сообщения

Каждому пользователю формируется только одно письмо, содержащее информацию по всем задачам, которые сменили статус в рамках мероприятия и к которым он имеет отношение.

·Тема сообщения:

Изменение статуса задачи;

·Текст сообщения:

½Новый статус: %название нового статуса задач%

Далее для каждой задачи выводится следующая информация:

½%Тип объекта % ID %debt_task.r_id%

½Название: %debt_task.name%

½Тип: %debt_task.typ (словарь 229)%

½Подтип: %debt_task.subtyp (словарь 230)%

½Дата завершения: %debt_task.dt_deadline%

½%пустая строка%

½%Информацию по следующей задаче%

где "Тип объекта" выводится в зависимости от debt_task.obj_id:

·8 – "Долг";

·46 – текст термина 255 для law_act.typ для law_act.id = debt_task.r_id;

·47 – "ИП".

·Пример сообщения:

½Долг ID 23567

½Название: Звонок

½Тип: Позвонить должнику

½Подтип: Позвонить настойчиво
