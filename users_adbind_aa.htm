Доменная авторизация

Если значение константы LoginUseADAuth = "Ложь", то работает текущий функционал проверки авторизации:

·если для текущего пользователя Windows существует соответствующий пользователь в таблице users (соответствие по win_user_name), то для пользователя Windows проверяются флаги блокировки учетной записи и истечения срока пароля; авторизация в Системе будет успешна, если у пользователя указанные флаги не выставлены, при этом автоматически пропускается окно ввода логина и пароля;

·если для текущего пользователя Windows не существует соответствующий пользователь в таблице users (соответствие по win_user_name), то авторизация идет по логину паролю в Системе. Пользователю отображается окно для ввода логина и пароля.

Если значение константы LoginUseADAuth = "Истина", то включается функционал принудительного запрашивания логина и пароля от доменной учетной записи.

·отключается пропуск окна ввода логина и пароля, при наличии соответствия текущему пользователю Windows записи в users;

·отключается операция смены пароля в Системе (кнопка "Сменить пароль" в окне авторизации в системе), как не применимая к механизму;

·пользователь может указать логин и пароль от любой другой учетной записи Windows;

·если пользователь указывает имя с доменом (в формате <домен>\<логин>), проверка пользователя осуществляется на указанном домене;

·если пользователь указывает имя без домена, в качестве домена используется домен текущего пользователя Windows;

·в Системе проверяется наличие соответствующего пользователя в системе Контакт (по users.win_user_name), проверка соответствия должна быть регистронезависимой, win_user_name должно быть заполнено с учетом домена (в формате <домен>\<логин>);

·дополнительно проверяется факт его блокировки users.block_flag;

·на контролере домена проверяется корректность логина и пароля, состояние учетной записи (блокирована, истек пароль);

·Если значение константы LoginADAuthGroupList не пусто, то дополнительно проверяется, что пользователь состоит в одной из групп, указанных в списке (проверка регистронезависимая);

·В случаях:

·отсутствия пользователя в системе Контакт;

·отсутствия пользователя на Домене;

·некорректной паре Имя пользователя-Пароль

в системе должно регистрироваться действие "Подбор пароля" (typ=1), в комментарии к которому должны быть указаны имя пользователя и домен;

·после каждой неудачной попытке нужно выставлять ожидание ввода в течении 10 секунд;

·после 3х неудачных попыток ввода должна блокироваться учетная запись контакт (users.block_flag), также как это реализовано в текущем варианте авторизации;

·в случаях:

·блокировки пользователя в системе Контакт;

·блокировки пользователя на домене или истечении срока действия его пароля;

·при не пустом значении константы LoginADAuthGroupList, если пользователь не состоит ни в одной из групп, перечисленных в константе,

необходимо выводить данную информацию пользователю, по аналогии с текущим механизмом авторизации.
