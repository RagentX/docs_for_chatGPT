Проверка данных

Для проверки данных в таблице нажмите кнопку Проверить в форме Загрузка данных. В процессе проверки данных в нижней строке формы будет отображаться тип операции (проверка данных), текущие данные о проценте выполнения работы, директория и имя файла.

Проверка данных включает:

·проверку на повтор преобразованных столбцов (могут повторяться только столбцы телефонов, адресов, текстовых адресов, поручителей должника, контактных и расчетных лиц);

·контроль наличия преобразованных столбцов для обязательных реквизитов. В перечень обязательных реквизитов входят:

·фамилия, имя и отчество персоны — для типов форматов "Досудебная работа", "Судебная работа", "Исполнительное производство" и "Досудебная работа (обновление)";

·название продукта, сумма, необходимая к погашению, дата выхода на просрочку — для типа форматов "Досудебная работа";

·тип судебной работы и размер требования по заявлению — для типа форматов "Судебная работа";

·общая сумма долга — для типа форматов "Исполнительное производство";

·контроль наличия значений по каждой записи для обязательных реквизитов;

·контроль корректности реквизитов даты и времени;

·контроль соотношения реквизитов даты и времени, который включает следующие проверки:

·дата рождения персоны не должна превышать дат выдачи кредита, выхода на просрочку, крайнего срока погашения и выдачи паспорта;

·дата выдачи кредита не должна превышать дат выхода на просрочку и крайнего срока погашения;

·дата выхода на просрочку не должна превышать крайнего срока погашения;

·контроль корректности числовых реквизитов;

·контроль корректности целочисленных реквизитов;

·контроль корректности словарных реквизитов, который включает в себя:

·контроль корректности целочисленного реквизита;

·контроль наличия значения реквизита в словаре;

·контроль корректности текстовых реквизитов, при котором проверяется, не превышает ли размер текста предельного значения для данного реквизита в базе данных;

·контроль корректности кодов KLADR;

·контроль и автоматическую корректировку названий регионов;

·контроль корректности названий портфелей;

·контроль наличия наименований городов по каждой записи в составе адресов;

·контроль соответствия областей, районов и городов заданным значениям констант (в жестком варианте проверки адресов);

·контроль наличия городов в справочнике KLADR (с учетом значений районов или областей в составе адресов);

·контроль наличия наименований улиц в составе адресов (только для улиц городов);

·контроль наличия улиц в справочнике ADDRESS_ZONE (только для улиц базовых городов — Москвы, Санкт-Петербурга, Екатеринбурга).

 

В перечень обязательных реквизитов входят:

·фамилия, имя и отчество персоны – для типов формата досудебная работа, судебная работа, исполнительное производство и досудебная работа (обновление);

·название продукта,  сумма, необходимая к погашению, дата выхода на просрочку – для досудебной работы;

·тип судебной работы и размер требования по заявлению – для судебной работы;

·общая сумма долга – для исполнительного производства;

·ФИО поручителя – для дозагрузки поручителей.

Примечание. В перечень обязательных реквизитов расчетных лиц помимо расчетного счета входят также БИК и банк, однако проверка наличия преобразованных столбцов и значений по каждой записи для этих реквизитов осуществляется позднее, в процессе загрузки данных.

Контроль соотношения реквизитов даты и времени включает следующие проверки:

·дата рождения персоны не должна превышать дат выдачи кредита, выхода на просрочку, крайнего срока погашения и  выдачи паспорта;

·дата выдачи кредита не должна превышать дат выхода на просрочку и крайнего срока погашения;

·дата выхода на просрочку не должна превышать крайнего срока погашения.

Контроль корректности словарного реквизита включает:

·контроль корректности целочисленного реквизита;

·контроль наличия значения реквизита в словаре.

При контроле корректности текстового реквизита проверяется, не превышает ли размер текста предельного значения для данного реквизита в базе данных.

Ошибкой первого рода при проверке данных таблицы является

·повторное описание какого-либо столбца, который не может повторяться;

·отсутствие описания столбца для какого-либо обязательного реквизита.

При наличии ошибки первого рода будет выдано одно из сообщений:

·"Некорректный формат. Повторное описание столбца <наименование столбца>";

·"Некорректный формат. Отсутствует описание столбца <наименование столбца>".

Ошибкой второго рода при проверке данных таблицы является

·отсутствие значения какого-либо обязательного реквизита в какой-либо записи;

·некорректное значение какого-либо реквизита даты и времени;

·некорректное соотношение какой-либо пары реквизитов даты и времени;

·некорректное значение какого-либо числового реквизита;

·некорректное значение какого-либо целочисленного реквизита;

·некорректное значение какого-либо текстового реквизита;

·некорректное название какого-либо портфеля.

При наличии ошибки второго рода (в отсутствии ошибок первого рода) будет выдано сообщение о первой ошибке второго рода "Загрузка невозможна. Ошибка в записи №<номер записи>".

Ошибкой третьего рода при проверке данных таблицы является

·несоответствие адреса заданным значениям констант в жестком варианте проверки (посторонняя область, район или город);

·отсутствие наименования области в справочнике KLADR;

·отсутствие наименования района в справочнике KLADR;

·пустое значение города в составе непустого адреса;

·отсутствие наименования города в справочнике KLADR (в области, в районе или в KLADR нет города);

·пустое значение улицы для одного из базовых городов;

·отсутствие наименования улицы базового города в справочнике ADDRESS_ZONE;

·некорректное значение кода KLADR;

·отсутствие кода KLADR в справочнике KLADR.

При наличии ошибки третьего рода (в отсутствии ошибок первого и второго рода) будет выдано сообщение о первой ошибке третьего рода "Загрузка невозможна. Ошибка в записи №номер".

Примечание. Если пользователь имеет право на сохранение текста адресов без ссылки на справочные данные, то ошибки третьего рода не фиксируются.

Если ошибки отсутствуют, по окончании проверки данных будет выдано сообщение "Данные соответствуют формату загрузки".

В отсутствие ошибок первого рода ячейки таблицы, по которым зафиксированы ошибки второго и третьего рода, будут окрашены (см. Цветовое отображение ошибок).

Повторная проверка данных производится после их ручного редактирования или редактирования с использованием сервисных режимов контекстного меню (см. Сервисные функции). При повторных проверках проверяются только данные, измененные пользователем. Если изменения отсутствуют, будет выдано сообщение: "Данные в ячейках таблицы не изменялись, повторная проверка отменена". При наличии данных, измененных вручную, производится их повторная проверка так же, как описано выше.
