Регистрация контакта с должником, поручителем или залогодателем

Show/Hide Hidden Text

Для того чтобы зарегистрировать результат контакта с должником, поручителем или залогодателем, выполните следующие действия:

1.В форме регистрации контакта путем последовательного перехода выберите результат в дереве результатов контактов. Для перехода по дереву нажимайте на следующие значки:

·значок , чтобы развернуть ветвь возможных исходов;

·значок , чтобы свернуть ветвь возможных исходов.

Примечание. Если значение константы RegContactCloseMode равно "Истина", то возможность выхода из модуля без регистрации исхода контакта определяется доступом REG_CONTACT_CLOSE.

2.Нажмите кнопку ОК или выполните двойной щелчок левой клавишей мыши по выбранному результату. Откроется форма ввода данных о результате контакта. Состав полей в форме и обязательность их заполнения зависит от настроек результата при добавлении результата контакта к дереву. Пример формы ввода информации для контакта типа "Звонок" с результатом "Подтверждение оплаты" приведен ниже.

Примечание: если в настройках результата контакта одновременно указана необходимость ввода платежа и необходимость ввода обещания, то при регистрации контакта с таким исходом сначала будет происходить регистрация неподтвержденного платежа, а затем регистрация введенного обещания. Это позволит избежать того, что новый платеж "погасит" новое обещание.

Примечание: если константа PromRepeatedControlAll имеет значение "Истина", то при наличии действующего обещания невозможно зарегистрировать новое обещание до истечения срока действующего обещания (как по долгу, так и по должнику) При вводе обещаний проверяется наличие активных обещаний. Если таковые есть, то новое обещание зарегистрировать нельзя. Также при включенной константе PromRepeatedControlAll можно добавлять обещания в день, равный обещанной дате.

Примечание: если константа PromRepeatedControlAll имеет значение "Ложь", но константа PromRepeatedControl имеет значение "Истина", то для операторв типа "Soft" при регистрации контакта типа "Звонок", "Выезд" или "Входящий звонок" также запрещено регистрировать обещание при наличии действующего обещания (как по долгу, так и по должнику).

Примечание. Если в настройках результата контакта указана необходимость ввода причины возникновения задолженности, то логирование ввода/изменения причины осуществляется следующим образом:

·Если константа RegContact.DebtReasonNotChange.CreateLog имеет значение "Истина", то логирование установки причины возникновения долга выполняется при каждой установке причины, в том числе и, если причина не изменилась.

·Если константа RegContact.DebtReasonNotChange.CreateLog имеет значение "Ложь", то установка причины возникновения долга логируется только в том случае, если причина изменилась.

Примечание. Если в настройках результата контакта одновременно указана необходимость ввода даты повторного звонка, то в поле Дата вы можете указать дату от текушей и до даты через максимально разрешенное количество дней. Максимально разрешенное количество дней  выбирается как меньшее из двух значений: значения константы RecallMaxDay и значением атрибута Кол-во дней до следующего звонка в настройке данного исхода контакта. Если вы указываете текущую дату, то время должно быть больше текущего времени. Константа RegContactRecallDayMode регулирует режим установки даты повторного звонка: если константа равна 0, то дата звонка указывается только для текущего долга, а если константа равна 1, то дата звонка указывается для всех долгов должника.

Если для долга есть активные акции, то форма будет содержать блок Предложить акцию. В поле Акция можно выбрать одну из доступных для данного долга активных акций, при этом поля Сумма к оплате, Сумма в руб., Дата окончания будут заполнены автоматически и недоступны для редактирования. При регистрации контакта для долга будет установлен флаг Акция предложена и автоматически заполнено поле Источник информации об акции в соответствии со словарем № 131.

3.В форме введите данные и нажмите кнопку ОК.

Примечание: если в настройках результата контакта указана необходимость ввода графика платежей, то введите график платежей, как описано в разделе Создание графика платежей. Если у должника уже есть график платежей, то будет выдано соответствующее сообщение и создание нового графика будет запрещено.

4.В некоторых случаях может открыться дополнительная форма ввода данных. Например, если в приведенной выше форме не указать ссылку на файл, подтверждающий оплату, появится форма для указания причины отсутствия файла. В этом случае заполните поля в открывшейся форме и нажмите кнопку ОК. Результат контакта будет зарегистрирован.

5.Если данный исход контакта содержит анкету для заполнения, то после указания всех остальных данных по нему откроется окно анкеты.

Согласно настройкам выбранного исхода контакта будет осуществлена смена статуса телефона (при регистрации контакта по телефону) или смена статуса адреса (при регистрации контакта по адресу).

Данные о зарегистрированном контакте будут доступны:

·В модуле Карточка должника на вкладке Контакты.

·В модуле Протокол контактов на вкладке Контакты.

Если контакт по телефону или адресу был зарегистрирован из каротчки поручителя или имущества, то соответствующий флаг Поручитель или Залогодатель будет включен.

Ввод значений произвольных атрибутов

Если в настройках результата контакта указана необходимость ввода значений произвольных атрибутов, то в форме ввода данных о результате контакта будет отображаться панель Произвольные атрибуты со списком атрибутов для заполнения. Обязательные для заполнения атрибуты обозначаются звездочкой.

Для того чтобы ввести значение произвольного атрибута:

1.Выберите атрибут из списка.

2.Введите или выбрите значение в поле ввода, расположенном под списком атрибутов.

3.Нажмите ENTER.

Реализовано сохранение названия колл-листа, в рамках которого был зарегистрирован контакт, а также ID кампании обзвона при создании записи о контакте в таблицу contact_log. 

Условия сохранения данных:

·Название колл-листа сохраняется в поле "call_list", если константа RegContact.CallList.Save имеет значение "Истина";

·ID кампании ЦОД сохраняется в поле "r_campaign_id", если константа RegContact.CampaignID.Save имеет значение "Истина".

Учет статистики по звонкам

При регистрации контакта с типом "Исходящий звонок" (исключая результаты автоинформатора) производится обновление в хранилище агрегированной информации по контактам. 

Зарегистрированный контакт учитывается в статистике "Количество контактов всего", если: 

·По контакту есть call_id (ID звонка):

Не было зарегистрировано любого контакта с аналогичным call_id по данному участнику договора. (если в рамках одного телефонного разговора будут зарегистрированы исходы по двум договорам, то это будут 2 различных факта по одному на каждый из договоров);

·По контакту нет call_id: 

По данному участнику договора не было зарегистрировано контакта аналогичным пользователем в течении определенного периода (задается константой).

Зарегистрированный контакт учитывается в статистике "Количество контактов, учитываемых в ограничении", если: 

·Для результата, по которому регистрируется контакт проставлен атрибут "Учитывать в ограничениях";

·Контакт был учтен в статистике "Количество контактов всего" (в рамках выполнения условий по call_id, описанных выше).

Для автоинформирования

Проверка ограничений коммуникации осуществляется на этапе формирования call-list.

Добавление коммуникации в статистику осуществляется при регистрации контакта с типом "Автоинформирование" для исходов, настроенных специально для регистрации контактов автоинформирования  (настраивается поддержкой).

Для кампании обзвона

Проверка ограничений коммуникации осуществляется на этапе формирования call-list.

Добавление коммуникации в статистику осуществляется при регистрации контакта.

Учет статистики по SMS

Статистика по смс должна обновляться (триггером) сразу после вставки сообщения в таблицу debt_sms. В статистике будут учитываться сообщения в следующих статусах: 

·Передано 

·Доставлено

·Заявка 

·В процессе отправки 

·Не известно

Если сообщение будет переведено в статус, отличный от описанных выше, то оно будет исключено из статистики, и наоборот, если сообщение перейдет в указанные статусы, то оно будет учитываться в статистике. 

В качестве даты сообщении для учета в статистике берется фактическая дата отправки (debt_sms.send_date), если эта дата не заполняется, то учитывается дата начала отправки (debt_sms.start_date).

В случае отложенной отправки сообщения, предполагаемые к отправке через некоторое время (в статусе "Готово к отправке") будут учитываться в статистике, соответствующей дню отправки. И такие сообщения будут учитываться при попытке поставить в очередь еще одно сообщение. 

 Обновление статистки по смс, которые изменили статус будет выполняться фоновым заданием. 

Учет статистики по Email

Статистика по email должна обновляться сразу после вставки сообщения в таблицу debt_email. В статистике будут учитываться сообщения в следующих статусах: 

·Заявка 

·Доставлено

Если сообщение будет переведено в статус, отличный от описанных выше, то оно будет исключено из статистики, и наоборот, если сообщение перейдет в указанные статусы, то оно будет учитываться в статистике. 

В качестве даты сообщении для учета в статистике берется фактическая дата отправки (debt_email.send_date), если эта дата не заполняется, то учитывается дата начала отправки (debt_email.start_date). Обновление статистки по email, которые изменили статус будет выполняться фоновым заданием.

Учет статистики по выездам

Зарегистрированный контакт с типом "Выезд" всегда учитывается в статистике "Количество контактов всего".

Зарегистрированный контакт с типом "Выезд" учитывается в статистике "Количество контактов, учитываемых в ограничении", если у узла дерева контактов проставлен атрибут "Учитывать в ограничениях" и "Перевести выезд в статус выполнения" и независимо от значения параметра "Считать результат успешным"
